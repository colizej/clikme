# ADR-001: Выбор Django как основного фреймворка

**Дата:** март 2026
**Статус:** Принято

## Контекст
Сайт clikme.ru работает на OpenCart (PHP) — платформе для e-commerce,
использованной для блога через плагин OptimBlog. Ограничения: сложность
кастомизации, ограниченный Python-стек, сложность автоматизации.

## Решение
Переход на **Django (Python 3.11+)**

## Причины
- Разработчик имеет опыт Django
- Гибкость для всех запланированных функций (блог, объявления, рассылки, бот)
- ORM, admin-панель, встроенная безопасность
- Большая экосистема пакетов
- Python → возможность ML/AI функций в будущем

## Последствия
- Требуется VPS (не shared hosting) — EU-сервер уже есть
- Миграция данных из MySQL → PostgreSQL через скрипт
- SEO-риск при переключении DNS — управляемый через 301-редиректы

# ADR-002: SQLite как основная БД

**Дата:** март 2026
**Статус:** Принято

## Решение
Использовать **SQLite** — встроенную БД Python.

## Причины
- Встроена в Python, нулевая инфраструктура
- Для сайта с 70-500 статьями и трафиком до 50k посетителей/месяц — полностью достаточна
- Не нужен отдельный сервер БД на VPS
- Бэкап = одна команда `cp db.sqlite3 db.sqlite3.bak`
- При необходимости миграция на PostgreSQL — один параметр в settings.py

## Ограничения
- Нет параллельных записей (не критично для контент-сайта)
- При росте до маркетплейса с транзакциями — рассмотреть PostgreSQL

## Миграция данных
OpenCart MySQL → SQLite: скрипт читает .sql файл через стандартный re/парсинг
или подключается к MySQL напрямую, записывает через Django ORM в SQLite.

# ADR-003: Кастомный User с AbstractUser с первого дня

**Дата:** март 2026
**Статус:** Принято

## Решение
Использовать **AbstractUser** (не стандартный auth.User).

## Причины
- Django не позволяет менять модель User после первой миграции без болезненного
  сброса БД
- Нам понадобятся поля: telegram_id, user_type, points (геймификация), avatar
- Лучше заложить сейчас с минимальными полями, чем потом переписывать

## Реализация
```python
# apps/users/models.py
class User(AbstractUser):
    pass  # минимум полей в начале, расширяем постепенно
```
В AUTH_USER_MODEL = 'users.User' — сразу в settings.

# ADR-004: Tailwind CSS CLI + минимум JS вместо SPA-фронтенда

**Дата:** март 2026
**Статус:** Принято

## Решение
**Tailwind CSS v4 CLI + HTMX** для интерактивности, без Vue/React/Bootstrap.

## Причины
- Tailwind CLI — без Node.js/npm на сервере, CSS собирается локально → один статический файл
- Utility-first = полный контроль над дизайном, никакого Bootstrap-look
- Минималистичный дизайн (Structura-стиль) проще реализовать в Tailwind
- Tailwind v4 — CSS-нативные переменные `@theme`, быстрее v3
- Один разработчик — нет смысла в отдельном фронтенд-стеке
- HTMX даёт динамику (бесконечная прокрутка, живой поиск) без JS-фреймворка
- Django-шаблоны — всё в одном месте

## Исключения
- Карты (Leaflet.js) — для каталога мест (Фаза 2)
- Загрузка изображений — можно Filepond.js
